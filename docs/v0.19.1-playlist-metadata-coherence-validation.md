# v0.19.1 Playlist Metadata Coherence Validation

## Scope

This beta corrects the playlist metadata pipeline without changing strict
recording resolution, reserve replenishment, ordering or playback
reconstruction. The permanent `track_id` and selected recording fingerprint
remain the identity boundary. Recording facts, release facts, user metadata
and diagnostic provenance are stored separately.

## Coherence rules

- Roon remains the authority for the playable selection, its artwork and its
  streaming or local source.
- A release is accepted only when its track list contains the selected title,
  artist and version family. When both are present, selected and release
  artwork must also match.
- Artist-catalog fallback may inspect only releases with the selected artwork;
  it cannot borrow an album from a different search candidate.
- MusicBrainz may add duration, ISRCs, composers, lyricists, genres and release
  years only after a Roon release or exact ISRC anchors the lookup.
- Multiple equally compatible recordings produce `metadata_status=conflict`.
  No popularity score is used to hide ambiguity.
- Every refresh replaces the previous catalog observation, so an invalid old
  album or duration cannot survive a failed verification.
- Legacy identities polluted by the previous enrichment implementation are
  reconstructed from their original `selected_candidate` while retaining the
  stored fingerprint.

Metadata status is independent from resolution status:

- `exact`: coherent release and complete core metadata;
- `partial`: verified recording or release with missing fields;
- `conflict`: incompatible release or more than one recording remains;
- `unverified`: no safe release or recording anchor was available.

Only `missing`, `ambiguous` or another identity-resolution failure affects the
reserve workflow. Partial or conflicting catalog metadata does not replace an
accepted song.

## Portal behavior

- Missing and ambiguous tracks use red and pastel-yellow icon-only warnings.
- Clicking either warning opens the existing per-track candidate search and
  explicit selection flow.
- Debug mode adds `Actualizar metadatos` to the playlist and `Reasignar
  canción` to every track detail.
- Track detail shows album/edition, duration, service/origin, composers,
  lyricists, genres, release years, track/disc, ISRC and version.
- `user_metadata` is rendered as a separate grid. Debug facts are tabulated
  and the complete technical payload is formatted in a collapsible JSON view.

## Automated validation

Run:

```powershell
pnpm run test
pnpm run build
node --check portal/app.js
```

Focused coverage verifies coherent release membership and artwork, safe
MusicBrainz conflicts, explicit version resolution, legacy identity cleanup,
metadata persistence during model-created playlist preflight, reserve behavior,
manual repair and portal controls.

## Post-deployment validation

1. Open `Rock clásico esencial` with Debug mode enabled and choose `Actualizar
   metadatos`.
2. Confirm progress visits every resolved track and the final notification
   separates verified, partial, conflict, unverified and failed counts.
3. Inspect `Riders On The Storm`: the album must either match its selected
   artwork/edition or remain visibly unverified; an unrelated LP must not be
   inferred.
4. Inspect `Won't Get Fooled Again (Remastered 2022)`: duration and edition
   must refer to the selected 2022 version, or the track must report a conflict.
5. Open a missing or ambiguous warning and explicitly select a candidate.
   Verify that enrichment runs after reassignment and that playback still
   reconstructs the chosen recording.
6. Create a playlist through MCP with primaries and reserves. Confirm accepted
   tracks contain `audio_metadata.metadata_status` and unresolved identities,
   not metadata gaps, are the only entries replaced by reserves.

After an explicit deployment, refresh the ChatGPT app and start a new
conversation so it loads the updated MCP tool description and instructions.
