# RoonIA v0.17.2 Beta Validation

Date: 2026-07-16
Status: phases 1 through 11 automated validation complete; local visual
validation complete through phase 6, focused phase 9 search/artist review and
focused phase 10 seek-clock review; user review pending.

## Phase 1 scope

- The portal header shows only the application version and appends `(beta)`
  when the configured update channel is beta.
- Build identifiers are no longer displayed in the header.
- Stale update status is no longer copied into the top bar.
- Portal action notifications use a single visible message for exactly three
  seconds; a newer message replaces the previous one immediately.
- Messages identify the affected zone, value, playlist, preset, user,
  credential or connection whenever that context is available.
- Success, informational, warning and error notices share the same restrained
  visual treatment and accessible live region.

## Phase 2 scope

- System settings present the installed version and build in a compact update
  block, appending `(beta)` when the selected channel is beta.
- The update-check action is labelled `Comprobar actualizaciones`.
- The available version and update action remain hidden until the update API
  confirms that a newer version or build exists.
- Persistent out-of-context labels such as `Sin comprobar`, `Al día`,
  `Completada` and `Sistema preparado` were removed from the System page.
- Active or failed update operations remain visible in context; completed
  operations continue to use the progress dialog without leaving stale state
  in System settings.
- Service restart is separated from software updates in its own `Servicio`
  panel.
- Roon status labels now describe the checked capability: core connection,
  playback control, music navigation and detected zones.

## Phase 3 scope

- Automatic update checks can be enabled or disabled independently from the
  network and channel settings; new installations enable them by default.
- When enabled, the service checks the selected channel immediately if no
  recent result exists and schedules the next check 24 hours later.
- The latest valid result is stored in `data/version-status.json` and restored
  only when it belongs to the currently installed version and build.
- A detected update produces a permanent `Nueva actualización disponible`
  notice below the header version and the existing detailed notice in System
  settings.
- The portal refreshes the header notice while open, and a manual check updates
  both locations immediately.

## Phase 4 scope

- Disabling beta updates opens a dedicated explanation with three outcomes:
  install stable now, retain the current beta until stable catches up, or
  cancel and continue receiving betas.
- The immediate option targets `main`, persists the stable channel and uses the
  existing update-progress flow, including the service restart.
- The deferred option persists the installed beta version and build, disables
  newer beta updates and shows its state permanently in System settings.
- During a deferred exit, `main` is checked daily even when the general
  automatic check toggle is off. A stable version behind the retained beta is
  not offered as an update; an equal or newer stable version triggers the
  authorized switch to `main` automatically.

## Phase 5 scope

- System settings contain a separate `Diagnóstico y soporte` panel with a
  persistent `Modo Debug` selector; it is not grouped with update controls.
- Normal mode hides the Registros tab, advanced OAuth diagnostics and the
  technical System detail grid.
- Debug mode reveals those surfaces immediately and adds environment, update
  channel, ports, MCP and Roon Browse capabilities, API authentication and the
  installed build to System.
- Normal-mode navigation rejects the hidden Registros destination and falls
  back to System. The setting affects portal visibility only and does not
  change `LOG_LEVEL`.

## Phase 6 scope

- Model-created playlists resolve all primary and reserve proposals before
  creating or replacing the persisted playlist.
- Every proposal requires title and artist credit. Confident album and year
  values help rank or validate a recording without being forced into the
  initial Roon search query.
- A temporary model-provided result ID cannot bypass title, credit, playable
  identity or recording-version validation.
- Standard and remastered recordings are accepted for a standard request;
  live, remix, cover, dub, acoustic and alternate versions require an explicit
  matching intent.
- Ambiguous, missing, duplicate and insufficiently enriched performance
  candidates are omitted and never persisted.
- Valid reserves fill rejected proposals. If the requested size remains short,
  two autonomous replenishment rounds are available without writing partial
  state. The second failed round saves a safe shorter playlist and reports the
  exact missing count.
- The final order prevents adjacent recordings by the same artist. Available
  Roon metadata and raw observations are stored separately from LLM hints and
  candidate provenance.
- Playback never reuses an expired Roon item key. It reconstructs a fresh
  reference from the stored identity and retries the original successful
  title-and-artist query when enriched credits or release context return no
  result; album and year remain ranking evidence rather than mandatory search
  text.

## Phase 7 scope

- Automatic playlist collages rotate every two seconds.
- A 2x2 or 3x3 collage changes one randomly selected position per cycle; a
  4x4 collage changes two randomly selected positions per cycle.
- Position selection uses a shuffled bag so it does not walk sequentially
  through the grid and avoids immediately reusing the last positions.
- Replacement artwork is loaded before the image source is changed, keeping
  the old tile visible on load failure and avoiding an intermediate black tile.
- The source change has no fade, scale, zoom or delayed transition.
- Track artwork keys are deduplicated before layout selection. Layout selection
  keeps a hidden reserve in one-tile collages when possible, and visible tiles
  remain unique whenever at least four distinct covers exist.

## Phase 8 scope

- Playlist cards show the playlist name first, followed by the number of songs
  and last playback, with the description last.
- Descriptions retain their four-line truncation but no longer reserve a fixed
  four-line height, eliminating inconsistent empty space below short text.

## Phase 9 scope

- Multi-type service searches execute in independent Roon sessions
  concurrently; the manual portal also starts its four category requests
  together and progressively renders isolated loading, result and error states.
- Empty loading sections use a compact circular activity indicator instead of
  rectangular result placeholders.
- The first response remains limited to 6 artists, 6 releases, 12 tracks and 6
  playlists. Larger Roon sections are indexed silently after first paint, up
  to 100 results, and `Ver más` reveals at most 12 entries per activation.
- `Mejor resultado` is rebuilt from the safe direct recommendation of each
  typed Roon response without adding another Browse request. Ambiguous or
  merely similar results never enter that block.
- A newer search aborts the previous request set and ignores stale responses.
- A still-valid exact catalog search reference is opened before a matching
  local-library identity, so artist fichas retain Roon's full classified
  streaming discography instead of collapsing to owned albums.
- Exact native-library artist and album identities are reopened through a
  ten-minute ordinal index whose cached item keys are never reused across
  sessions.
- Artist details never infer an official discography from a global name-only
  album search. Album details never infer an ordered tracklist from shared
  artwork or assign sequential numbers to unordered search matches.
- Detail responses and portal fichas expose origin, completeness, ordering and
  identity verification. Related results remain visibly separate from an
  official tracklist.
- Streaming album rows inside an exact Roon contents/Tracks list are retained
  even when Roon omits per-row track numbers or durations; complete pagination,
  not optional row metadata, determines whether the list is complete.
- Indistinguishable exact candidates require manual selection, and slash band
  names such as AC/DC remain intact.

## Phase 10 scope

- Releasing the mini-player seek control immediately anchors its local clock to
  the selected position while playback continues.
- The mini-player render signature includes Roon's `seek_position`, so later
  zone refreshes reconcile the progress bar and elapsed-time label with the
  authoritative playback position.
- A failed seek invalidates the optimistic render state and restores the
  position reported by Roon on the next refresh.

## Phase 11 scope

- Playlist summaries include the combined known track duration in hours and
  minutes next to the song count.
- The backend calculates duration across the complete playlist rather than the
  limited artwork page returned to the portal.
- A total is exact when every track has duration metadata; otherwise the portal
  labels the known lower bound as `al menos` instead of understating it as exact.
