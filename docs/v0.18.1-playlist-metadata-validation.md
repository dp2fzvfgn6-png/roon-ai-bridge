# v0.18.1 Playlist Metadata And Repair Validation

## Scope

This beta adds a playlist metadata enrichment boundary and a repair
orchestrator without changing the established primary, reserve, replenishment
or finalization rules used by playlist builds.

## Implemented Behavior

- Accepted build candidates are enriched before persistence.
- Sparse Roon observations no longer overwrite stored metadata with nulls.
- Album detail can supply album, duration, year, track/disc number, artwork,
  source and quality when Roon exposes them.
- Sparse track search results are opened in Roon and followed through the
  `Go to Album` action so the matching album row can supply album, duration,
  release year and track/disc position.
- When a track action does not expose its album, enrichment searches the
  resolved artist catalog and correlates safe equivalent track artwork with a
  release before reading its ordered track list.
- When Roon still omits duration or release year, the separate recording
  metadata service queries MusicBrainz using the already-identified title,
  artist and album. It can only fill empty metadata fields and cannot replace
  the selected Roon recording.
- Metadata completeness is tracked separately from recording resolution.
- `roon_refresh_playlist_metadata` refreshes incomplete, selected or all
  resolved playlist tracks.
- `roon_resolve_playlist` repairs missing/ambiguous tracks and accepts explicit
  model selections.
- Portal playlist rows show a pastel-yellow warning for `ambiguous` and the
  interface red for `missing`.
- The warning opens an editable search, explains the failure, supports an
  automatic retry and allows the user to select one playable Roon track.
- Debug mode exposes a full-playlist metadata refresh action. The portal sends
  one bounded track refresh at a time and shows progress instead of holding a
  single silent request open for the whole playlist.
- Duplicate refreshes of the same playlist track share the active Roon
  operation, preventing repeated clicks from competing for browse sessions.

## Preserved Invariants

- Missing or ambiguous identity can consume a reserve; incomplete metadata
  cannot.
- Build sessions still allow two replenishment rounds and persist only when
  finalized.
- Manual selections must reference a currently playable Roon track.
- Roon result identifiers remain temporary; the stored track identity remains
  the durable reconstruction key.
- No SQLite schema migration is required. Enrichment diagnostics are stored
  with the existing resolution metadata.

## HTTP Operations

- `POST /playlists/:playlist_id/metadata/refresh`
- `POST /playlists/:playlist_id/tracks/:track_id/repair`
- `GET /playlists/:playlist_id/tracks/:track_id/candidates`
- `POST /playlists/:playlist_id/tracks/:track_id/match`

## Automated Validation

Run before publishing:

```powershell
pnpm run test
pnpm run build
node --check portal/app.js
git diff --check
```

Focused coverage verifies non-destructive metadata merge, album-track
hydration from both album and sparse track results, duplicate refresh
coalescing, manual repair provenance, reserve behavior, MCP schemas and build
session continuity, plus the portal repair and progress controls.

Final local result: 221 tests passed, TypeScript build passed, portal JavaScript
syntax passed and `git diff --check` passed.

## Manual Validation After An Explicit Deployment

1. Refresh the ChatGPT app connection and start a new conversation so the new
   MCP tool catalog is loaded.
2. Create a disposable playlist through `roon_save_playlist`; confirm accepted
   tracks contain album and duration whenever Roon exposes them.
3. Run `roon_refresh_playlist_metadata` and confirm identity status is not
   downgraded when only metadata remains partial.
4. Open an ambiguous or missing playlist row in the portal, verify its warning
   colour and explanation, search candidates and select the intended recording.
5. Enable debug mode and run the playlist metadata refresh action. Confirm the
   button advances through the playlist and the album/duration columns update.
6. Verify the resulting playlist and logs. Playback is not required for these
   checks and no zone volume should be changed.
