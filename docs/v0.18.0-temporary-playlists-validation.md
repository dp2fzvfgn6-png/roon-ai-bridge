# v0.18.0 Temporary Playlists Validation

## Scope

This change adds disposable working playlists for LLM requests such as music
for cooking, cleaning or concentrating, without mixing those lists into the
user's saved collection. It also completes deep media reads for playlists
selected from the native Roon catalog.

## Behavioral contract

- `roon_create_temporary_playlist` uses the existing strict candidate preflight,
  reserve selection and two-round replenishment flow.
- Each completed temporary playlist has a stable ID and its own `expires_at`.
- Ordinary playlist listings default to saved playlists. Temporary lists are
  available only through the explicit temporary scope/tool.
- Expired playlists, tracks and custom cover files are deleted automatically
  before playlist reads or mutations.
- Promotion is in place: it preserves the playlist ID, tracks, order and cover,
  while optional name and description changes are atomic with lifecycle removal.
- `temporary_playlist_expiry_days` defaults to 7, accepts 1-365 and affects only
  lists created after the preference is saved.
- The Portal System panel always exposes the expiry preference. The dedicated
  temporary-playlist manager is rendered only when Debug mode is enabled and
  supports the same management actions as a saved playlist plus promotion.
- `roon_get_media_entity` returns ordered, paginated tracks for a selected Roon
  catalog playlist. `count` is bounded to 100, `offset` selects the page and
  `pagination.has_more` tells the model when the source is complete.
- Catalog track references remain temporary; persistent `track_id` and identity
  fingerprints are assigned only after tracks are saved in RoonIA.
- Batched additions omit recordings already present in the target by persistent
  fingerprint or normalized title/artist identity.

## Automated validation

Focused tests cover:

- saved/temporary listing isolation and in-place promotion;
- expiration cleanup with cascading track deletion;
- temporary build purpose across replenishment rounds;
- default, validation and persistence of the expiry preference;
- MCP manifest registration and model sequencing instructions;
- a 175-track catalog playlist split into pages of 100 and 75, preserving
  positions and normalized title, artist and album metadata;
- gateway forwarding of playlist `count` and `offset`;
- duplicate lookup against tracks already stored in the destination;
- Portal markup, Debug visibility hooks, API wiring and preference persistence.

Run before release:

```powershell
pnpm run test
pnpm run build
git diff --check
```

No LXC deployment or audible Roon operation is part of this local validation.

Local validation result: 197 tests passed, TypeScript build passed, portal
JavaScript syntax check passed and `git diff --check` reported no errors.
